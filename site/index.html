What it is

`bd-console` reads `.beads/issues.jsonl` for issue lists, shells out to `bd` for live comments and writes, and renders markdown docs from the target repo. It is designed for repos that already use beads and want a simple local dashboard without introducing a frontend framework or build pipeline.

Prerequisites

  • Node 18 or newer
  • `bd` installed and available on `PATH`
  • A target repo with `.beads/` already initialized
cd ~/code/my-project
bd where
bd export -o .beads/issues.jsonl

Quick start

Choose one install path:

# clone the repo somewhere local
git clone https://github.com/formless63/bd-console.git ~/code/bd-console

# or install the CLI globally from GitHub
npm install -g git+https://github.com/formless63/bd-console.git

Then run the guided setup against the repo you actually want to inspect:

# from a clone
node ~/code/bd-console/scripts/init.mjs --repo ~/code/my-project --apply-agent-docs

# or with a global install
bd-console-init --repo ~/code/my-project --apply-agent-docs

The init flow refreshes `.beads/issues.jsonl`, writes `bd-console.json` if needed, and can update `AGENTS.md` / `CLAUDE.md`.

bd-console now runs as a Global Hub. Start one instance, and register your projects to it.

bd-console start
bd-console add ~/code/my-project
bd-console add .

Then open http://localhost:4180 to view the Hub Dashboard.

You can manage the daemon with:

bd-console status
bd-console stop

Configuration

The init script writes `bd-console.json` for you unless one already exists. Create or edit it manually only when you need overrides.

{
  "port": 4180,
  "host": "127.0.0.1",
  "docRoots": ["docs", ".planning"],
  "token": "shared-secret"
}
  • host: defaults to `127.0.0.1`; expose wider only on purpose
  • port: defaults to `4180`
  • docRoots: narrows markdown discovery in large repos
  • token: gates write actions while leaving reads open

Environment overrides: `BD_CONSOLE_REPO`, `BD_CONSOLE_HOST`, `BD_CONSOLE_PORT`, `BD_CONSOLE_TOKEN`.

Security posture

  • Default bind host is `127.0.0.1`.
  • Writes are open unless a token is configured.
  • If you bind off localhost, set a token.
  • Issue IDs are validated before write commands.
  • Writes use `execFile` with an args array, not shell strings.

For LAN or tailnet access, keep the setup explicit: 

node serve.mjs --repo ~/code/my-project --host 0.0.0.0 --port 4180
# and set BD_CONSOLE_TOKEN or token in bd-console.json

Export freshness

Comments are live through `bd comments --json`. The issue list is exported JSONL. `bd-console` now refreshes the export automatically when it is missing or older than `.beads/last-touched`, and the UI shows sync state in the top bar.

You still want explicit refreshes when operating outside the dashboard, especially in scripted or agent-driven flows: 

bd create "something"
bd export -o .beads/issues.jsonl

Docs discovery

Without `docRoots`, the server auto-discovers markdown under the repo root, skipping directories like `.git`, `node_modules`, `.next`, `dist`, and `.beads`. The `.planning` directory is explicitly allowed.

In large repos, define `docRoots` so users only see the documentation that matters.

Ideas and triage conventions

  • Quick capture creates a `task` with a default `triage` label.
  • Treat `triage` as an inbox, not a final classification.
  • For doc-derived ideas, add a `doc:<path>` label so provenance survives triage.

Keyboard shortcuts

`bd-console` supports high-speed triage through hotkeys designed to let you keep your hands on the keyboard. These shortcuts are disabled when typing inside input elements.

Shortcut Action
j Select the next issue in the list
k Select the previous issue in the list
/ Focus search query text box
c Focus comment edit field in the active details pane
i Open the Capture an idea modal window
Esc Close capture modal or remove current active focus

Agent setup

The init script can update agent docs for you:

bd-console-init --repo ~/code/my-project --apply-agent-docs
bd-console-init --repo ~/code/my-project --apply-agent-docs --create-missing-agent-docs

The inserted guidance includes:

  • how to start `bd-console`
  • that `.beads/issues.jsonl` should be refreshed after non-UI `bd` mutations
  • the `triage` / `doc:<path>` conventions
  • how to verify the dashboard is reachable
bd export -o .beads/issues.jsonl
curl -I http://127.0.0.1:4180/api/meta

Verification

npm run check
npm run smoke
npm_config_cache=/tmp/bd-console-npm-cache npm pack --dry-run

The smoke test creates a temporary beads repo, starts the server, exercises the API, and covers inline issue edits, comments, and quick capture.

Troubleshooting

`Error: no beads database found`
Run inside a beads repo, pass `--repo`, or inspect `bd where`.
Issue list looks stale
Run `bd export -o .beads/issues.jsonl` and refresh the page.
Comments work but status changes do not appear
Comments are live; issue list data comes from the exported JSONL.
Docs tree is noisy
Set `docRoots` explicitly in `bd-console.json`.
Token prompts keep appearing
Check the configured token value and clear `localStorage.bd_token` if needed.

Publishing this site

This repo ships the docs site as plain static files under `site/`.

  • GitHub Pages: workflow at `.github/workflows/docs-site.yml` deploys `site/` on pushes to `main`.
  • Cloudflare Pages: set build command to none and output directory to `site`.

Local preview:

npm run docs:serve
