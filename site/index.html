What it is

`bd-console` is a Global Hub: one server per machine, with the repos you work in registered into it. It reads `.beads/issues.jsonl` for issue lists, shells out to `bd` for live comments and writes, renders markdown docs from each registered repo, and — on the same host — lets you see tmux sessions and schedule prompts to be typed into them later. It is designed for repos that already use beads and want a dashboard without introducing a build pipeline; the frontend is a no-build app with all its dependencies vendored, not pulled from npm at install time.

Prerequisites

  • Node 22 or newer (required — the prompt scheduler uses `node:sqlite`)
  • `bd` installed and available on `PATH`
  • optionally, `tmux` on the same host, for the tmux/scheduler features
  • each repo you register needs `.beads/` already initialized
cd ~/code/my-project
bd where
bd export -o .beads/issues.jsonl

Quick start

Choose one install path:

# clone the repo somewhere local
git clone https://github.com/formless63/bd-console.git ~/code/bd-console

# or install the CLI globally from GitHub
npm install -g git+https://github.com/formless63/bd-console.git

Then run the guided setup against the repo you actually want to inspect:

# from a clone
node ~/code/bd-console/scripts/init.mjs --repo ~/code/my-project --apply-agent-docs

# or with a global install
bd-console-init --repo ~/code/my-project --apply-agent-docs

The init flow refreshes `.beads/issues.jsonl`, writes `bd-console.json` if needed, and can update `AGENTS.md` / `CLAUDE.md`.

bd-console now runs as a Global Hub. Start one instance, and register your projects to it.

bd-console start
bd-console add ~/code/my-project
bd-console add .

Then open http://localhost:4180 to view the Hub Dashboard.

You can manage the daemon with:

bd-console status
bd-console stop

There is no per-repo server mode anymore — `bd-console` always runs as one hub, and `--repo` only exists on `bd-console-init`/`scripts/init.mjs` to pick which repo to set up and register. See the project's `docs/upgrading.md` if you're coming from an older single-repo install.

Hub, tmux & scheduler

Beyond issues and docs, the hub also exposes host-level (not per-repo) views:

  • tmux sessions — see every tmux session/pane running on the hub's host, with a live scrollback preview.
  • Prompt scheduler — queue a literal prompt to be typed into an already-running, named tmux session at a future time. It never creates sessions and never runs a command like `claude -p` itself; a job against a session that isn't there when it fires ends up `failed`, not silently dropped.

Both degrade gracefully: without `tmux` installed, the tmux view reports no sessions available; without Node 22's `node:sqlite`, the scheduler endpoints return `501` and the rest of the hub keeps working.

Configuration

Host, port, token, and persist are global settings for the hub — they live in `~/.config/bd-console/config.json`, not in any per-repo file. Manage them with: 

bd-console settings list
bd-console settings set host 0.0.0.0
bd-console settings set port 4180
bd-console settings set token a-shared-secret
bd-console settings unset token
bd-console settings              # interactive walkthrough

A per-repo `bd-console.json` (the init script writes one for you) still exists, but its job shrank to a single key: 

{
  "docRoots": ["docs", ".planning"]
}
  • docRoots: narrows markdown discovery in large repos; omit to auto-discover

Any `host`/`port`/`token` left over in an old per-repo `bd-console.json` is now ignored.

Environment overrides: `BD_CONSOLE_HOST`, `BD_CONSOLE_PORT`, `BD_CONSOLE_TOKEN`, `BD_CONSOLE_PERSIST`, `BD_CONSOLE_CONFIG_DIR`, `BD_CONSOLE_SCHED_INTERVAL`.

Security posture

The default bind is `0.0.0.0:4180`. Because that's more permissive than binding only to localhost, the very first run on a terminal walks you through a short setup instead of applying it silently:

  • This machine / home LAN — bind `0.0.0.0`, open writes recommended.
  • VPS / internet-exposed — bind a private or tailnet interface first; a random token is generated only as a last resort if you bind `0.0.0.0` anyway.

If a box is ever exposed publicly, put an authenticating reverse proxy (e.g. Pangolin) in front rather than relying on the token alone — the token only gates write endpoints against casual use.

  • Writes are open unless a token is configured.
  • Issue IDs and tmux session names are format-validated before every write.
  • Writes use `execFile` with an args array, not shell strings.
  • tmux pane previews are token-gated the same as writes, since pane contents can hold secrets.

Change any of this later with `bd-console settings`.

Export freshness

Comments are live through `bd comments --json`. The issue list is exported JSONL. `bd-console` now refreshes the export automatically when it is missing or older than `.beads/last-touched`, and the UI shows sync state in the top bar.

You still want explicit refreshes when operating outside the dashboard, especially in scripted or agent-driven flows: 

bd create "something"
bd export -o .beads/issues.jsonl

Docs discovery

Without `docRoots`, the server auto-discovers markdown under the repo root, skipping directories like `.git`, `node_modules`, `.next`, `dist`, and `.beads`. The `.planning` directory is explicitly allowed.

In large repos, define `docRoots` so users only see the documentation that matters.

Ideas and triage conventions

  • Quick capture creates a `task` with a default `triage` label.
  • Treat `triage` as an inbox, not a final classification.
  • For doc-derived ideas, add a `doc:<path>` label so provenance survives triage.

Keyboard shortcuts

`bd-console` supports high-speed triage through hotkeys designed to let you keep your hands on the keyboard. These shortcuts are disabled when typing inside input elements.

Shortcut Action
j Select the next issue in the list
k Select the previous issue in the list
/ Focus search query text box
c Focus comment edit field in the active details pane
i Open the Capture an idea modal window
Esc Close capture modal or remove current active focus

Agent setup

The init script can update agent docs for you:

bd-console-init --repo ~/code/my-project --apply-agent-docs
bd-console-init --repo ~/code/my-project --apply-agent-docs --create-missing-agent-docs

The inserted guidance includes:

  • how to start `bd-console`
  • that `.beads/issues.jsonl` should be refreshed after non-UI `bd` mutations
  • the `triage` / `doc:<path>` conventions
  • how to verify the dashboard is reachable
bd export -o .beads/issues.jsonl
curl -I http://127.0.0.1:4180/api/meta

Verification

npm run check
npm run smoke
npm_config_cache=/tmp/bd-console-npm-cache npm pack --dry-run

The smoke test creates a temporary beads repo, starts the server, exercises the API, and covers inline issue edits, comments, and quick capture.

Troubleshooting

`no .beads/ found at <path>` from `bd-console add`
Run `bd init` in that repo first, or point `add` at the right path.
A registered project 404s or doesn't show up
Confirm it's actually registered with `bd-console list`; re-add it if not.
`start` says it started but the dashboard doesn't load
Run `bd-console status` — it checks real port reachability, not just whether a pid is alive. `start` also tails the log automatically on failure.
Issue list looks stale
Run `bd export -o .beads/issues.jsonl` and refresh the page (the server also auto-refreshes it when stale).
Comments work but status changes do not appear
Comments are live; issue list data comes from the exported JSONL.
Docs tree is noisy
Set `docRoots` explicitly in that project's `bd-console.json`.
Token prompts keep appearing
Check the configured token value and clear `localStorage.bd_token` if needed.
The tmux or scheduler API returns `501`
You're on Node <22 (`node:sqlite` isn't available). Upgrade Node.

Publishing this site

This repo ships the docs site as plain static files under `site/`.

  • GitHub Pages: workflow at `.github/workflows/docs-site.yml` deploys `site/` on pushes to `main`.
  • Cloudflare Pages: set build command to none and output directory to `site`.

Local preview:

npm run docs:serve
